00012 /* lexicographical sorting of numbers
00013 
00014 We wish to be able to sort a report on an alphanumeric column and for the numbers to be sorted numerically and the non-numerics to be sorted alphabetically eg 2,10,A,B. We expect the classic pick behaviour and not be forced into alphabetic sorting of numbers eg 10,2.
00015 
00016 If possible we also wish mixed alphanumerics to be sorted reasonably eg A2,A10 (ie not A10,A2).
00017 
00018 NB the following is based on 32 bit ints and floats but the same techniques can be applied to 64 bit doubles, mapping them to some 64-bit ints like long long int.
00019 
00020 long int http://en.wikipedia.org/wiki/Long_integer
00021 long long int is from C99 and is required to be at least 64 bits
00022 
00023 http://www.cygnus-software.com/papers/comparingfloats/comparingfloats.htm
00024 
00025 The IEEE float and double formats were designed so that the numbers are lexicographically ordered, which in the words
00026 of IEEE architect William Kahan means if two floating-point numbers in the same format are ordered ( say x < y ),
00027 then they are ordered the same way when their bits are reinterpreted as Sign-Magnitude integers.
00028 
00029 This means that if we take two floats in memory, interpret their bit pattern as integers, and compare them, we can tell
00030 which is larger, without doing a floating point comparison. In the C/C++ language this comparison looks like this:
00031 
00032  if (*(int*)&f1 < *(int*)&f2)
00033 
00034 This charming syntax means take the address of f1, treat it as an integer pointer, and dereference it. All those
00035 pointer operations look expensive, but they basically all cancel out and just mean treat f1 as an integer. Since we
00036 apply the same syntax to f2 the whole line means compare f1 and f2, using their in-memory representations interpreted
00037 as integers instead of floats.
00038 
00039 Kahan says that we can compare them if we interpret them as sign-magnitude integers. Thats unfortunate because most
00040 processors these days use twos-complement integers. Effectively this means that the comparison only works if one or
00041 more of the floats is positive. If both floats are negative then the sense of the comparison is reversed the result
00042 will be the opposite of the equivalent float comparison. Later we will see that there is a handy technique for dealing
00043 with this inconvenience.
00044 
00045     int aInt = *(int*)&A;
00046     // Make aInt lexicographically ordered as a twos-complement int     (or 0x8000000000000000 for double)
00047     if (aInt < 0)
00048       aInt = 0x80000000 - aInt;
00049 
00050 Compiler issues
00051 
00052 In our last version of AlmostEqualUlps we use pointers and casting to tell the compiler to treat the in-memory
00053 representation of a float as an int. There are a couple of things that can go wrong with this. One risk is that int
00054 and float might not be the same size. A float should be 32 bits, but an int can be almost any size. This is certainly
00055 something to be aware of, but every modern compiler that I am aware of has 32-bit ints. If your compiler has ints of
00056 a different size, find a 32-bit integral type and use it instead.
00057 
00058 Another complication comes from aliasing optimizations. Strictly speaking the C/C++ standard says that the compiler
00059 can assume that different types do not overlap in memory (with a few exceptions such as char pointers). For instance,
00060 it is allowed to assume that a pointer to an int and a pointer to a float do not point to overlapping memory. This
00061 opens up lots of worthwhile optimizations, but for code that violates this rulewhich is quite commonit leads to
00062 undefined results. In particular, some versions of g++ default to very strict aliasing rules, and dont like
00063 the techniques used in AlmostEqualUlps.
00064 
00065 Luckily g++ knows that there will be a problem, and it gives this warning:
00066 
00067  warning: dereferencing type-punned pointer will break strict-aliasing rules
00068 
00069 There are two possible solutions if you encounter this problem. Turn off the strict aliasing option using the
00070 -fno-strict-aliasing switch, or use a union between a float and an int to implement the reinterpretation of a float
00071 as an int. The documentation for -fstrict-aliasing gives more information.
00072

00092 /* strategy
00093 
00094 The problem isnt just about sorting 10A and 2B correctly, it is about correctly sorting simple cases like 10, 2, A, and B as
00095 2, 10, A, B ie mixed data but not mixed within any one field. All picks can do that because they have a type-less comparison
00096 operator > that works "correctly" on text and textual numbers.
00097 
00098 To avoid requiring the implementation of a proper pick comparison operator in every sql database that exodus uses in future a
00099 function is provided that preprocesses a bit of text into a text that, when compared as ascii/unicode, produces the right result.
00100 Essentially this involves replacing all the numbers found in a string with a 16 hex digit representation of the number. This function
00101 must be called around all sorting and indexing fields and also on all filtering values passed like 'ABC' in "with xyz > 'ABC'".
00102 Since exodus build up the sql queries before each selection then the function can be built into the sql command. Note that selection
00103 on indexes built with natural order function in them must have the function matching in field and data when using the index.
00104 
00105 Note that in the conversion to 16 hex digits that all surplus zeroes are lost losing informaition. A proper comparison operator
00106 would be able to sort 001 before 1 but the current strategy cannot allow for this.
00107 
00108 NB all pick databases do text to floating point conversions on the fly since they store numbers as strings. This conversion-to-natural-order routine does a floating point conversion (but maybe for performance it should use the cpu's fp unit to do the job)
00109 The comparison then doesnt require an additional conversion to fp so the performance is perhaps ok for sorting pure numbers.
00110 
00111 Trailing E for exponent not supported at the moment.
00112 
00113 2       <       10
00114 .10     <       .2
00115 -20     <       -1
00116 
00117 2.0     =       2
00118 02      =       2
00119 
00120 ABC     <       BC
00121 
00122 Flat 2B <       Flat 10A
00123 
00124 A-2     <       A-10            dots and dashes within text don’t imply negative/decimal points
00125 A -10   <       A -2        unless prefixed by space
00126 A.2     <       A.10
00127 A .10   <       A .2
00128 
00129 A1.10   =       A1.1            excess zeros are ignored in sorting
00130 A010  = A10
00131 A1.0  = A1
00132 
00133 A1-2    <       A01-10  multi-level number sort
00134

00139         assert(sizeof(double) == sizeof(long long int));

00141         //view the double as an 64 bit integer
00142         long long int integer1=*(long long int*) & double1;
00143 
00144         if (integer1<0)
00145                 //invert the negative numbers and clear the high bit so that
00146                 //lexicographically they sort in the right order and before the positive numbers
00147                 integer1=0x8000000000000000LL-integer1+0x8000000000000000LL;

00149                 //set the high bit so that positive numbers appear lexicographically after the negative numbers
00150                 integer1-=0x8000000000000000LL;

00153                 //put a 9 which sorts the same as 0 and removes the need for fill/width formatting below
